### BREAKTHROUGH: Nested Closure Root Cause Found (T109)
**Bug:** In the WASM binary, `fn(x) { fn(y) { puts(y); fn(z) { z } } }`:
1. The outer function creates a closure with table_index=0, but index 0 points to fn(z) not fn(y)
2. env_ptr=0 means x is NOT captured into the closure environment
3. Table elem mapping: `(elem (i32.const 0) func 16 15)` — inner function at 0, middle at 1

**Root cause:** When a direct function (non-capturing) compiles nested function literals, 
the table indices are assigned in compilation order (inner first, middle second) but the closure 
creation code in the outer function uses the first index (0) thinking it's the middle function.

**Fix approach:** Either (a) fix table index assignment order, or (b) ensure the closure references 
the correct function index. The compileFunctionLiteral returns and the outer function should use 
that return's table index.

**Learning:** WASM disassembly (wasm2wat) is invaluable for debugging code generation bugs. 
Installed WABT for future use.

### T229 (BUILD): Major JIT fix - guard exit stack corruption
**Root cause analysis:**

- Block side trace recording for guards with stack values
- Added irStack remapping in DCE, CSE, renumbering passes
