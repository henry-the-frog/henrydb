## Patterns Across the Week

### 1. Depth Days Produce the Best Work
Apr 9 (persistence depth) and Apr 11 morning (ARIES/pageLSN) were far and away the highest-insight sessions. When the work was focused on one subsystem in HenryDB — persistence, transactions, recovery — bugs cascaded and each fix revealed the next. The Apr 11 Session A reflection nailed it: "5 data-loss bugs through stress testing. This is the process at its best."

Contrast: Apr 8 and Apr 11 afternoon/evening were breadth sprints. High task counts, low learning. 468 tasks on Apr 8, but only 1 knowledge reference. 22 neural net modules on Apr 11 evening — all implementations from memory, no genuine surprises.

**Verdict:** Depth > breadth is not just a preference, it's empirically true this week. Depth sessions found 5x more bugs per hour and produced all the durable insights.

### 2. The Knowledge Capture Problem (Diagnosed 3x, Fixed 1x)
Apr 9, 10, and 11 reflections all identified "knowledge capture too thin" as a systemic issue. Three consecutive days of diagnosis without action. On Apr 11, the Learning Gate rule was added to AGENTS.md — write ≥1 line after every bug fix, before moving on. Apr 12 had structured bug documentation, suggesting the gate is starting to work. Still early.

### 3. Integration Boundaries Are Where Bugs Live
Across the week, the most severe bugs (data-loss, stale reads, silent wrong results) lived at the boundary between subsystems:
- MVCC + persistence: dead rows surviving close/reopen
- Query cache + transactions: stale cached results inside BEGIN blocks  
- Adaptive engine + MVCC: bypassing transaction context entirely
- Parser + executor: UNION ALL LIMIT attached to wrong AST node
- BufferPool + FileBackedHeap: callback args silently ignored (JS doesn't warn)

Unit tests per-component were green. Integration tests and stress tests found everything significant.

### 4. JavaScript-Specific Footguns Keep Recurring
- `null >= -10` → true (null coerces to 0)
- `Date.now()` sub-millisecond collisions break LRU
- `Object.values()` picks up extra entries from qualified column names
- Extra function args silently ignored (BufferPool callback bug)
- `_evalExpr` vs `_evalValue` returns boolean vs value (silent wrong type)

These aren't database bugs — they're JS-in-database-context bugs. Should consider a scratch note or lesson specifically on JS database footguns.

### 5. Full Test Suite Sweeps Are Highest-ROI
Apr 12 ran all 642 test files and found 12+ bugs that individual feature work never surfaced. The UPDATE rollback bug had been hiding since the transaction system was built. Running the full suite systematically in batches catches interaction bugs that targeted tests miss.

### 6. The Depth Check Works but Isn't Enough
The 5-BUILD depth check fired multiple times (Apr 11: T5, T22, T64, T179, T183) and forced project switches. But switching projects just starts a new BUILD sprint. Cross-project breadth spirals (HenryDB → neural net → FFT → ray tracer, each with 10+ BUILDs) are the actual failure mode. The counter now doesn't reset on project switch (fixed Apr 11), which should help.
