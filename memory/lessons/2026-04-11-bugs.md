# Bugs Found 2026-04-11 (Session B)

## 1. Conv2D Double Batch-Size Division (neural-net)

**Symptom**: Conv2D learned N² times slower than expected. CNN classifier achieved only 18% accuracy on trivial pattern detection.

**Root Cause**: `Conv2D.backward()` normalized gradients by dividing by batchSize. Then `Conv2D.update()` divided by batchSize *again*. Dense layers only divide once (in update), so there was an inconsistency.

**Fix**: Removed the batchSize division from `Conv2D.update()` since backward already handles it.

**Lesson**: When multiple layers have different gradient normalization conventions, **numerical gradient checking catches it**. The gradient checker verified correctness per-layer but the full training pipeline revealed the issue. Always test end-to-end training, not just individual layer gradients.

**Prevention**: Standardize the gradient convention: either ALL layers normalize in backward, or ALL normalize in update. Don't mix.

## 2. UNION ALL + LIMIT Not Applied (henrydb)

**Symptom**: `SELECT ... UNION ALL SELECT ... LIMIT 10` returned all rows, ignoring LIMIT.

**Root Cause**: The SQL parser's `parseSelect()` consumes ORDER BY and LIMIT. When parsing `... UNION ALL SELECT ... LIMIT 10`, the LIMIT was consumed by the right SELECT's parser, not the UNION node. The UNION executor didn't apply LIMIT.

**Fix**: After parsing the UNION's right query, strip ORDER BY/LIMIT from it and attach them to the UNION AST node. The UNION executor now applies these to the combined result.

**Lesson**: SQL syntax is ambiguous about whether ORDER BY/LIMIT belongs to the last SELECT or the whole UNION. Standard SQL says it belongs to the whole query. When adding new compound query operators, always test their interaction with ORDER BY/LIMIT.

## 3. CTE Alias Resolution (henrydb)

**Symptom**: `WITH ds AS (...) SELECT ds.avg_sal FROM ds` returned undefined.

**Root Cause**: CTE materialized rows only had unqualified column names (`avg_sal`), not alias-qualified names (`ds.avg_sal`). The resolver couldn't find qualified names.

**Fix**: When materializing CTE/view rows, add alias-qualified entries (`alias.column_name`) alongside the unqualified ones.

**Lesson**: SQL column resolution is surprisingly complex. Qualified names (table.column) need to work alongside unqualified names (column) in all contexts: CTEs, views, subqueries, JOINs. Test both forms.

## 4. Recursive CTE with 3+ Columns (henrydb)

**Symptom**: `WITH RECURSIVE fib AS (SELECT 1 AS n, 1 AS val, 0 AS prev UNION ALL SELECT n+1, val+prev, val FROM fib WHERE n < 10)` produced garbage data (val=true, n=null).

**Root Cause**: Two bugs compounding:
1. Expressions like `n+1` and `val+prev` both got the default column name "expr", causing key collision.
2. The fix attempt used `_evalExpr()` (for boolean conditions) instead of `_evalValue()` (for arithmetic). `_evalExpr` returns `true` for unhandled types like `arith`.

**Fix**: Skip the named column mapping. Instead, directly evaluate each recursive SELECT column expression against the source row using `_evalValue()`, mapping them positionally to the base case's column names.

**Lesson**: The `_evalExpr` vs `_evalValue` distinction is a landmine. Boolean context (`WHERE`, `ON`, `HAVING`) uses `_evalExpr`. Value context (`SELECT`, `SET`, `INSERT`) uses `_evalValue`. Mixing them produces silent wrong results. Consider merging into a single evaluator that returns the appropriate type.

---

## Meta-Lesson: Stress Testing Finds Bugs That Unit Tests Don't

All 4 bugs were found through **stress testing** or **gradient checking**, not through unit tests. Unit tests verify individual features in isolation. Stress tests exercise feature interactions that unit tests miss:

- Gradient checking: mathematical verification of the full backward chain
- Optimizer stress tests: complex multi-feature SQL queries
- Training convergence tests: end-to-end pipelines that expose gradient issues

**Rule**: After adding a feature, write at least one stress test that combines it with 2+ other features.
