- 08:50 BUILD T7: CRITICAL ACID BUG in HenryDB. BEGIN never assigned _currentTxId → all DML auto-committed to WAL → crash recovery replayed uncommitted data. Root cause: transaction lifecycle (BEGIN/COMMIT/ROLLBACK) was implemented at the savepoint level but not wired to the WAL. The WAL's txId-based commit tracking was correct, but BEGIN never provided a txId. This is a design-level integration bug — two subsystems (savepoints for rollback, WAL for durability) were built independently and never connected. 10 tests written, 5 exposed the bug. Learning: when two subsystems share a concept (transaction identity), trace the data flow between them.
- 09:15 BUILD T11: 5th neural-net bug — MoE batch training divergence. Dense.backward() overwrites dWeights per-sample. When processing batch samples in a loop through shared experts, only last sample's gradient survived. Learning: any module that processes batch elements in a loop through shared sub-modules has this vulnerability. The gradient check (which I wrote earlier) passed because it uses very small batch sizes and the input gradient was correct — only weight gradients were wrong.
- 09:55 EXPLORE: SAT solver SMT bug — _processAssertion() silently skipped string-format assertions (Array.isArray returns false for strings). ALL string constraints ignored → checkSat() always returned SAT regardless of formula. Root cause: assert() stores raw strings but _processAssertion() expected parsed arrays. Integration gap: the parser (parseSmtExpr) existed but wasn't called in the processing pipeline. Same pattern as HenryDB ACID bug — two subsystems (parser + solver) not connected.
- 10:07 BUILD T22: HenryDB GROUP BY + window function bug. _selectWithGroupBy() never called _computeWindowFunctions(), so RANK() OVER (...) combined with GROUP BY silently dropped the window column. Also aggregate ORDER BY in OVER clause crashed (object passed where string expected). Integration gap: window function pipeline and GROUP BY pipeline were built independently and never connected.
