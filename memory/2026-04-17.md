## Session B (2:15 PM onward)
### T145-T147: VACUUM+MVCC Depth Testing
- **3 bugs found and fixed:**
  1. **Result cache + MVCC visibility bug** — Session reads with snapshot isolation populated the shared result cache (db.js), causing subsequent auto-commit reads to see stale data. Root cause: cache write had no txId guard. Fix: only cache when `_currentTxId === 0`.
  2. **Same root cause** manifested in DELETE+re-INSERT+VACUUM scenario.  
  3. **VACUUM + crash recovery data loss** — VACUUM disabled WAL during physical deletes (pageLSN didn't advance). On recovery, stale pageLSN caused full page clear+replay, resurrecting vacuum'd rows (10→5→10 after crash). Fix: keep WAL enabled during VACUUM physical deletes. Also renamed `_physicalizeDeletesNoWal` → `_physicalizeDeletes`.
- 19 depth tests added covering: concurrent writes, version chain cleanup, overlapping snapshots, SSI, auto-vacuum thresholds, crash recovery, edge cases
- All 19 new tests passing, 230/232 related tests pass (2 pre-existing failures in vacuum.test.js)
- **Learning:** The result cache is a shared global resource that must be transaction-aware. Any caching layer that sits below MVCC must either be keyed by transaction/snapshot or only used outside transactions.
- **Learning:** WAL-skipping optimizations (even for "internal" operations like VACUUM) can silently corrupt crash recovery by keeping pageLSN stale. The write-ahead protocol means ALL physical page modifications must advance the LSN.

### T150-T157: Constraint + Crash Recovery Depth Testing
- **3 more bugs found and fixed (6 total for Session B):**
  4. **PRIMARY KEY uniqueness not enforced on INSERT** — The UNIQUE constraint check only tested `schema[ci].unique`, ignoring `schema[ci].primaryKey`. Single-column PKs had no uniqueness enforcement at all!
  5. **CHECK constraint not enforced on UPDATE** — `_validateConstraints()` was only called in the INSERT path. UPDATE could violate CHECK (e.g., set balance to -50).
  6. **NOT NULL not enforced on UPDATE** — Same root cause: `_validateConstraints` missing from UPDATE path.
- Fix for UPDATE: call `_validateConstraints` between heap delete and insert, with rollback-on-failure
- Also documented: UNIQUE + concurrent SI is a known limitation (needs unique index locks, like PostgreSQL)
- 21 depth tests added covering constraints after crash recovery, FK+VACUUM, trigger side-effects
- **Learning:** Constraint enforcement must be called on ALL write paths (INSERT, UPDATE, UPSERT). Only checking one path leaves others wide open. Same pattern as the result cache bug — shared invariants enforced in only one code path.

### T165-T169: Neural-net Optimizer Correctness
- **1 more bug (15 total Session B, 37 today):**
  15. **Adam weight decay was decoupled (AdamW-style) instead of L2** — `param -= wd*lr*param` is decoupled (AdamW). Correct Adam: `effGrad = grad + wd*param` (L2 through gradient, goes through adaptive rate). Both Adam and AdamW produced identical results — should be different.
- 20 optimizer tests: step-by-step reference verification for SGD, Momentum, Adam, RMSProp, AdamW
- All 5 optimizers converge on simple quadratic
- **Learning:** Weight decay vs L2 regularization is a subtle but important distinction. Adam-L2 couples decay with adaptive rate (different effective decay for each parameter). AdamW decouples (uniform decay). The original Adam paper uses L2 but many implementations accidentally use decoupled.

### Overall Day Summary (April 17)
This was the deepest depth day ever. Session A found 22 bugs at integration boundaries. Session B found 16 more at optimization fast-paths and coverage gaps. The meta-patterns:
1. **Fast paths bypass invariants** — result cache, index scans, UPSERT all had "optimized" paths that skipped MVCC or constraint checks

3. **WAL skipping = crash recovery bugs** — Any optimization that bypasses WAL (VACUUM, ALTER TABLE) creates data loss after crash
4. **Serialization code rots** — fromJSON only handled 1/12 layer types, Conv2D toJSON referenced wrong properties
