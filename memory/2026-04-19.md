### Learning capture
- **SERIAL + UNIQUE constraint interaction**: The INSERT handler had a pre-_insertRow UNIQUE check that used raw column-list-ordered values. This is fundamentally wrong when columns are specified in a different order than the schema, or when SERIAL columns are omitted. The fix: use `_orderValues()` to map to schema order before checking, and skip SERIAL columns that haven't been assigned yet.
- **Dual SERIAL mechanisms**: db.js has TWO serial increment mechanisms — `_serialCounters` (checks `type === 'SERIAL'`) and sequences (checks `serial` property). Only the sequence-based one fires because the CREATE TABLE handler stores type as 'INT' with a `serial` property name, not type 'SERIAL'. The _serialCounters path is dead code for SERIAL columns created via CREATE TABLE.

### T150 BUILD — MVCC ROLLBACK TO SAVEPOINT fix (MAJOR)
Root cause: UPDATE undo closure used "scorched earth" approach — delete ALL rows, re-insert snapshot. This destroyed all other transaction state during ROLLBACK TO SAVEPOINT.

New approach: targeted undo tracking each UPDATE's specific changes:
1. Pre-UPDATE snapshot captures physical row keys + alive status
2. Undo only deletes rows NOT in pre-UPDATE physical row set
3. xmax restoration only for rows alive before UPDATE
4. Re-insertion only for snapshot rows no longer physically present

Also fixed `_snapshotUpdateRows` — was capturing pre-transaction state (skipping tx-created rows) instead of pre-UPDATE state (all visible rows).

**Learning: closure scope in undo logs** — The undo closures captured the pre-transaction snapshot (excluding rows created by the tx) rather than the pre-UPDATE snapshot. For full ROLLBACK this was fine; for savepoint ROLLBACK TO it was catastrophically wrong because it overwrote intermediate transaction state.
