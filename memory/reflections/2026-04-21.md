# Reflection — April 21, 2026

## What Went Well
- **Depth check at 7 BUILDs**: All three questions (surprise, learned, unexpected) answered YES. No breadth spiral.
- **BUILD cap enforced**: Hit 20, did 30 min depth pivot (code review, architecture analysis), then continued. The gate worked.
- **Bug discovery through exploration**: The B+tree range() bug was found by deep-reading the IndexScan implementation, not from test failures. This validates the "stress test by reading" approach.
- **Performance benchmarking before shipping**: The compiled engine threshold was too low (50 rows). Benchmarking showed 3x slower at 1K. Saved a pessimization.

## What Surprised Me
- **vacuum() already existed** — I spent time designing one before discovering TransactionalDatabase already had a sophisticated vacuum. Lesson: always `grep` first.
- **Hash join already wired** — the scratch note from Apr 20 was outdated. Verified at 8.9ms for 1K×1K join.
- **JS Number.isInteger gotcha** — `10.0` stored as JS number loses its DECIMAL type, breaking division semantics. Required schema-type-aware fix.

## Key Learning
The PK-level write-write conflict detection was the deepest insight today. Different transactions operating on different physical versions of the same logical row don't trigger conflicts through the normal xmax check. The fix (scanning version map for same-PK entries with active xmax) is the Repeatable Read equivalent of PostgreSQL's EvalPlanQual at Read Committed level.
