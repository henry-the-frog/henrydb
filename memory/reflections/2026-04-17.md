## What We Learned

### The "Every Fast Path Is a Bypass" Pattern
The single most important insight: **every performance optimization or shortcut is a potential correctness bypass.** When you add a fast path (result cache, index scan, direct heap access, WAL-skipping), you create a code path that might skip a correctness check (MVCC visibility, constraint validation, crash recovery logging). 

Examples found today:
- Result cache: bypassed MVCC visibility → stale reads
- B-tree index scan: bypassed MVCC → deleted rows visible
- UPSERT: bypassed _validateConstraints → CHECK/NOT NULL/FK skipped
- VACUUM: bypassed WAL → crash recovery data loss
- _physicalizeDeletesNoWal: bypassed WAL → same crash issue

### The "Write Path Coverage" Pattern
If you enforce an invariant (constraints, MVCC, WAL logging) on ONE write path (INSERT), you must enforce it on ALL write paths (UPDATE, UPSERT, MERGE, FK CASCADE, ALTER TABLE). Today we found 5 write paths that were missing constraint validation.

### Testing Strategy
The highest-ROI testing approach: **test at the INTERSECTION of two features.** VACUUM alone is correct. MVCC alone is correct. VACUUM + MVCC has 3 bugs. This applies universally: constraint + UPDATE, index + MVCC, serialization + new layer types.

## Session B Addendum (5:30 PM–8:00 PM)

### The Session Within a Session
Session B went DEEP on ALTER TABLE WAL recovery and found it was a 14-bug iceberg. What started as "WAL-log ALTER TABLE" turned into fixing the entire crash recovery architecture for DDL operations. This is the best kind of depth work.

### The "Layered Architecture Wiring" Bug Pattern
HenryDB has 3 layers: TransactionalDatabase → Database → FileWAL. DDL operations cross all layers but each layer assumed the layer above handled persistence. The fix required patching `logDDL` onto the inner Database's WAL, adding a DDL replay phase to crash recovery, handling table rename cascading effects (physical file, heap, DiskManager, versionMap, table object), and treating txId=0 as always-committed.

**Lesson:** When you have N abstraction layers, every feature that crosses layers needs explicit wiring tests at each boundary.

### MVCC Is Genuinely Solid
42 MVCC tests across window functions, subqueries, JOINs, GROUP BY, HAVING, DISTINCT, aggregates, write skew, phantom reads — all correct. The MVCC implementation is the strongest part of HenryDB. The bugs are all at the persistence/DDL boundary, not in the core MVCC logic.

Session B alone: 14 bugs fixed across WAL recovery (4), crash recovery (3), checkpoint boundary (3), parser (1), CTE duplicate columns (1), view persistence (1), PK NULL acceptance (1). Plus 1 known limitation documented (ALTER TABLE backfill + checkpoint).
