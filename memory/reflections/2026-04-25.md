# Reflection: 2026-04-25 (Session A — Midpoint)

## What Went Well
1. **Rapid bug fixing**: Fixed 7 bugs (string truncation, WAL syncMode, EXPLAIN cost, LIMIT subquery, TRUE/FALSE, REGEXP, SUM/AVG coercion) in the first 20 BUILDs. Each fix was targeted and tested.
2. **Differential fuzzer**: Created in the last BUILD slot. Found 2 new bugs in 5 minutes. This is the single highest-ROI tool for HenryDB quality.
3. **Deep exploration**: The BUILD cap forced me into EXPLORE mode, which revealed that both HenryDB and monkey-lang are far more sophisticated than the TODOs suggested. monkey-lang is a mini-V8 with 20K lines of compiler/runtime. HenryDB has 6 execution engines.
4. **Documentation**: Blog post, neural-net tutorial, 8 new scratch notes, MEMORY.md update.

## What Could Be Better
1. **TODO list accuracy**: Many "TODO" items (GENERATED columns, CROSS APPLY, INLJ) were already implemented. Should verify before adding.
2. **Fuzzer should have been built earlier**: The differential fuzzer found bugs that no amount of manual testing would have found quickly.
3. **BUILD cap timing**: Hit 20 BUILDs faster than expected because many were quick. The per-task quality was fine but the count accumulated fast.

## Key Insights
1. **Analysis ≠ Application**: Both projects have sophisticated analysis infrastructure (SSA, escape analysis, selectivity estimation) that isn't wired into the execution path. The gap between "having the data" and "using the data" is where the value is.
2. **Silent failures > crashes**: The scariest bugs are silent ones (PAGE_SIZE data loss, SUM concatenation instead of addition). They don't announce themselves.
3. **Systematic testing > manual testing**: The fuzzer (200 random queries) found things that 8172 hand-written tests missed.
