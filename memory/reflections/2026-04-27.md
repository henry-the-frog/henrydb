# Reflections — 2026-04-27 Session A

## What Worked Extremely Well

### WASM Performance Optimization Arc
The session told a clear story: start with a working but unoptimized WASM backend, then systematically identify and eliminate bottlenecks. Each optimization built on the last:
1. Wire WASM into benchmarks (baseline: 3.5x VM)
2. Pre-compilation (4.7x VM — eliminate recompile overhead)
3. `knownInt` type inference (36.7x VM — eliminate host import crossings)
4. Return type inference (fib 35% faster — recursive calls stay in WASM)

The biggest lesson: **WASM host import crossings are the dominant cost**. A single `call __add` (WASM→JS→WASM) is 10-100x slower than a native `i32_add`. Type inference that eliminates these crossings has outsized impact.

### Bug Discovery Through Feature Work
The int/pointer confusion bug was found while adding HashDestructuringLet — a seemingly simple feature that revealed a fundamental design flaw in the untagged value system. This is exactly the kind of discovery that makes feature work valuable even when the features themselves are minor.

## Key Technical Insights

1. **Untagged values are fundamentally fragile**: Integer values that happen to be valid memory addresses get misinterpreted as pointers. The data-segment offset trick (65536) is a band-aid. NaN-boxing or WASM GC types are the proper fix.

2. **Return type inference unlocks recursive optimization**: A simple 3-iteration fixed-point analysis can determine that `fib` returns integers, enabling all recursive additions to use native `i32_add`.

3. **Node.js v22 supports WASM GC and EH natively**: struct/array types compile without flags. Exception handling (try/catch/throw) works. This means monkey-lang can use advanced WASM features.

4. **The transpiler's advantage is V8 TurboFan inlining**: WASM can't speculate or inline across function boundaries. For recursive workloads, V8's speculative JIT wins. For tight loops, WASM's guaranteed optimization wins.
