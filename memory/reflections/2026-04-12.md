But that's exactly the breadth momentum the cap exists to prevent. Blog posts consume time that could go to stress testing.

## One concrete improvement

The full test suite sweep is the highest-ROI activity we've found (12+ bugs in one session). It's also the activity most likely to trigger genuine depth work (cascade debugging). Right now it's ad-hoc — it happens when someone thinks to do it.

**Fix: Sweep Gate rule.** Every second session of the day (Session B, or the first session after a break >1 hour) MUST start with a full test suite sweep of the active project before any new feature work. The sweep identifies what's actually broken; feature work can only start after the sweep is clean or all found issues are triaged.

This pairs with the depth check: the depth check governs pace within a session, the sweep gate governs what a session STARTS with. Together they should prevent the pattern of "depth day that's actually a feature factory."

## Change Made

Added Sweep Gate convention to AGENTS.md under Work Quality, after the Learning Gate:

```
### Sweep Gate (second+ sessions per day)
Every second session of the day (or any session after a break >1 hour) MUST start with a full test suite sweep of the active project before any new feature work begins. Run all tests, triage what's broken, fix or file everything found. Only then start building. The sweep is the session's opening move, not an afterthought.
```

Also moved blog cap from MEMORY.md-only to AGENTS.md conventions so it's in the critical path.
