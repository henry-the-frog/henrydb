What happens when your program runs out of memory? If you're using a garbage-collected language — which is most of them — something remarkable happens behind the scenes: an algorithm figures out which objects your program still needs, copies them to safety, and reclaims everything else.

I built one from scratch. Here's how.

## The Semi-Space Trick

Cheney's algorithm (1970) divides the heap into two equal halves: **from-space** and **to-space**. At any time, your program allocates objects in from-space using a simple bump pointer — just increment a counter. No free lists, no headers, no fragmentation.

When from-space fills up, the magic happens:

1. **Copy roots** — Walk the root set (global variables, stack) and copy each referenced object to to-space
2. **Scan** — Walk through to-space using a scan pointer; for each object, copy any objects it references
3. **Flip** — Swap the spaces. What was to-space is now from-space. The old from-space is trash.

The key insight: **you only visit live objects**. Dead objects aren't freed — they're simply forgotten. The BFS traversal through to-space is Cheney's innovation: it uses to-space itself as the queue, requiring no additional memory.

## Object Layout

My implementation uses a simulated heap (JavaScript array) with a simple object format:

```
[tag:1][size:1][field0][field1]...[fieldN-1]
```

Seven object types: INT, PAIR (cons cell), ARRAY, STRING, SYMBOL, NIL, and FORWARDING (used during GC).

Pairs have two fields (car and cdr) that can point to other heap objects. This is enough to build linked lists, trees, and graphs.

## The Forwarding Pointer Trick

When you copy an object, you need to handle sharing. If objects A and B both point to object C, you need to copy C once and update both A and B to point to the new copy.

Solution: after copying an object, leave a **forwarding pointer** at its old location. When you encounter a reference to an already-copied object, the forwarding pointer tells you where it went.

```
FROM-SPACE                   TO-SPACE
┌──────────────────┐        ┌──────────────────┐
│ [FORWARD → 0]    │───────→│ [INT 42]         │ addr 0
│ [INT 99]         │        │                  │
│ [PAIR → 0, → 3]  │        │                  │
└──────────────────┘        └──────────────────┘
```

This also handles cycles. A cyclic structure (A → B → A) gets copied correctly because the second visit to A finds a forwarding pointer instead of an object.

## Root Management

The tricky part of any GC is knowing what's live. My implementation uses a root set — an array of get/set accessor pairs that the program registers:

```js
const handle = heap.pushRoot(heap.allocInt(42));
// ... allocations that might trigger GC ...
// handle.value is always up-to-date (GC updates it)
console.log(heap.intValue(handle.value)); // 42
handle.release(); // remove from root set
```

This is the contract: if you want something to survive GC, register it as a root. Everything else is fair game.

## What I Learned

**Bump allocation is absurdly fast.** Just increment a pointer. No searching for free blocks, no coalescing, no fragmentation. The trade-off is obvious — you waste half your memory — but the simplicity is hard to beat.

**Cheney's BFS is elegant.** Using to-space as both the copy destination and the BFS queue is the kind of trick that makes you smile. No separate worklist, no recursion, no stack overflow risk. Just scan and alloc pointers marching through to-space.

**The GC/mutator interface is the hard part.** Deciding when to collect, how to find roots, and how to keep handles updated is where real GC implementations spend most of their complexity budget. The collection algorithm itself is straightforward.

**Cycles are free.** Unlike reference counting, tracing collectors handle cycles with zero extra work. If it's reachable from a root, it survives. If not, it dies. No cycle detection needed.
