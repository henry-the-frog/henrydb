# Lessons

## 2026-04-18: Feature Capability Detection
**Pattern**: `if (obj.method) { optimize() }` with no else clause
**Failure**: HOT chain detection set isHotUpdate=true, checked heap.addHotChain, didn't exist, silently skipped BOTH chain creation AND index updates. Rows became invisible.
**Fix**: `if (canOptimize && hasCapability) { optimize } else { fallback_to_correct_behavior }`
**Takeaway**: Optional enhancements must always have an explicit fallback. Silent "oh well" is dangerous.

## 2026-04-18: Result Cache Invalidation on DDL
**Pattern**: SQL result cache returns stale results after CREATE OR REPLACE FUNCTION
**Failure**: Function body changed but cache still had old query results
**Fix**: Clear cache on any function/procedure CREATE or DROP
**Takeaway**: Any operation that changes query semantics must invalidate caches

## 2026-04-18: Verify Assumptions Against Code
**Pattern**: Gap analysis said HenryDB lacked PG-style in-progress transaction list
**Reality**: Code already had snap.activeSet — equivalent to PG's xip[]
**Takeaway**: Always read the code before planning improvements. Saved building something that already existed.

## 2026-04-18: TCO Pattern Unity
**Insight**: All three TCO variants (self-recursive, general named, closure) use the same pattern:
1. Compile arguments to temp locations
2. Set up calling convention (a-regs)
3. Emit epilogue to tear down current frame
4. Jump (not call) to target
The only difference is the jump target. This simplicity surprised me.
